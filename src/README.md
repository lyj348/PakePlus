# AI人工智能互动式播报系统 - 党建阵地知识问答

## 功能概述

本系统是基于Qwen3/DeepSeek大语言模型的党建阵地知识问答系统，具有以下特点：

- ✅ 基于Qwen3/DeepSeek大语言模型
- ✅ 专注于党建阵地相关知识问答
- ✅ 支持不少于10轮上下文的对话能力（实际设置为20轮）
- ✅ 智能上下文管理，保持对话连贯性
- ✅ 美观的现代化UI界面
- ✅ 支持模型切换（Qwen3/DeepSeek）
- ✅ 对话记录导出功能

## 功能特性

### 1. 对话功能
- 支持多轮对话，系统会自动维护上下文
- 上下文管理支持至少10轮对话（实际设置为20轮）
- 智能对话历史管理，自动优化上下文长度

### 2. 知识领域
系统专注于党建阵地相关知识，包括：
- 党的基本理论
- 党史和党的优良传统
- 党的组织建设
- 党的思想建设
- 党的作风建设
- 党的纪律建设
- 党建阵地建设实践
- 党员的权利和义务
- 党的组织原则和制度

### 3. 模型支持
- **Qwen3**: 阿里云通义千问大语言模型
- **DeepSeek**: DeepSeek大语言模型
- 支持在界面中实时切换模型

### 4. 用户界面
- 现代化渐变设计
- 响应式布局，支持移动端
- 实时对话显示
- 输入状态指示
- 对话轮次统计

## 使用方法

### 1. 直接使用（模拟模式）

直接打开 `index.html` 文件即可使用。当前版本使用模拟API响应，适合演示和测试。

### 2. 配置真实API（生产环境）

要使用真实的Qwen3或DeepSeek API，需要修改 `chat.js` 文件中的API配置：

```javascript
// 在 chat.js 的 APIManager 类中
this.apiConfig = {
  qwen3: {
    endpoint: 'https://api.example.com/v1/chat/completions', // 替换为真实端点
    apiKey: 'your-qwen3-api-key', // 替换为真实API密钥
    model: 'qwen-plus'
  },
  deepseek: {
    endpoint: 'https://api.deepseek.com/v1/chat/completions', // 替换为真实端点
    apiKey: 'your-deepseek-api-key', // 替换为真实API密钥
    model: 'deepseek-chat'
  }
};
```

然后取消注释 `callLLM` 方法中的真实API调用代码，注释掉模拟响应代码。

### 3. API调用示例

真实API调用格式：

```javascript
const response = await fetch(config.endpoint, {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${config.apiKey}`
  },
  body: JSON.stringify({
    model: config.model,
    messages: messages, // 包含系统提示和对话历史的完整消息列表
    temperature: 0.7,
    max_tokens: 2000
  })
});
```

## 技术实现

### 上下文管理

系统使用 `ChatContext` 类管理对话上下文：

- 自动维护系统提示词
- 保存用户和助手的对话历史
- 智能截断，保持最近20轮对话
- 自动统计对话轮次

### API集成

`APIManager` 类负责API调用：

- 支持多模型切换
- 统一的API调用接口
- 错误处理和重试机制
- 模拟模式用于测试

### UI管理

`UIManager` 类管理用户界面：

- 消息显示和格式化
- 输入框自动调整
- 滚动控制
- 状态指示

## 注意事项

1. **API密钥安全**: 在生产环境中，不要将API密钥直接写在代码中，建议使用环境变量或配置文件
2. **跨域问题**: 如果API端点与前端不在同一域名，需要配置CORS
3. **错误处理**: 系统已包含基本的错误处理，但建议根据实际API响应进行优化
4. **性能优化**: 对于大量对话历史，可以考虑使用后端服务进行上下文管理
